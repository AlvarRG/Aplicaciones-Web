Descripción Detallada del Proyecto Bistro FDI

1. Introducción y Visión del Proyecto

El proyecto Bistro FDI consiste en la especificación, diseño, prototipado e implementación completa de una aplicación web para la gestión integral del restaurante de la Facultad de Informática. Este desarrollo no se limita a una simple página informativa, sino que constituye un sistema de información complejo diseñado para digitalizar el flujo de trabajo de un establecimiento de hostelería moderno.

Modernización del Servicio

Tomando como referencia la experiencia de usuario de las grandes cadenas de restauración rápida, Bistro FDI busca eliminar los cuellos de botella tradicionales: las largas colas en barra y la incertidumbre sobre el tiempo de espera. La aplicación centraliza la interacción cliente-restaurante, permitiendo que cualquier usuario con un smartphone o portátil pueda consultar la carta en tiempo real, personalizar su pedido y realizar el pago sin necesidad de interacción física inmediata.

Optimización Operativa y Logística

Desde la perspectiva interna, Bistro FDI actúa como un ERP (Enterprise Resource Planning) ligero para el personal de la cafetería. El sistema coordina la comunicación entre la sala (camareros) y los fogones (cocineros), sustituyendo los tickets de papel por comandas digitales que se actualizan en tiempo real. Esto permite organizar la preparación de los platos por orden de llegada y prioridad, reduciendo los tiempos de servicio y minimizando el desperdicio de alimentos mediante una gestión precisa del stock y las categorías de productos.

El proyecto se desarrollará de manera incremental, empleando tecnologías web estándar y siguiendo buenas prácticas de ingeniería del software para garantizar la escalabilidad, la seguridad en la gestión de usuarios y la robustez en las transacciones de pedidos.

2. Tipos de Usuarios y Roles

La aplicación gestiona cuatro roles diferenciados. Es importante destacar que los roles son jerárquicos (de menos a más prioridad): las acciones que puede hacer un usuario incluyen las de su rol y todas las de los roles con menor prioridad.

  • Cliente: Consumidor final. Puede registrarse, gestionar su perfil, realizar pedidos y consultar su estado, así como beneficiarse del programa de fidelización.
  • Camarero: Personal de sala. Se encarga de entregar los pedidos preparados (verificando que estén completos, incluyendo bebidas) y realizar el cobro presencial si no se ha pagado online.
  • Cocinero: Personal de cocina. Se encarga de seleccionar los pedidos pagados para cocinarlos y notificar cuando los platos están listos.
  • Gerente: Administrador del sistema. Se encarga de gestionar el catálogo (altas, bajas y modificaciones de productos y categorías), las ofertas y las recompensas.

3. Funcionalidades del Sistema

A continuación se detallan las 4 funcionalidades clave que desarrollaremos en esta página web:

Funcionalidad 0: Gestión de Usuarios

Permite el registro, login y gestión de perfiles. Se almacena nombre de usuario (único), email, nombre completo, contraseña y rol.

Destaca la gestión de Avatares, permitiendo al usuario elegir entre imágenes predefinidas, subir una foto propia o usar uno por defecto. El sistema de roles permite al Gerente ascender o cambiar el rol de cualquier usuario.

Funcionalidad 1: Gestión de Productos y Categorías

El Gerente administra la carta.

  • Categorías: Nombre, descripción e imagen opcional.
  • Productos: Incluyen nombre, categoría, imágenes, disponibilidad (stock) y estado de oferta.
  • Precios e IVA: El sistema gestiona el precio base y aplica el tipo de IVA correspondiente (4%, 10%, 21%) para mostrar el precio final automáticamente.
  • Borrado: Se implementa un borrado lógico (flag "ofertado") para no perder histórico.

Funcionalidad 2: Gestión de Pedidos

Es el núcleo de la aplicación, adaptado a dispositivos móviles y escritorio. Un pedido atraviesa los siguientes estados:

  1. Nuevo: En creación (carrito).
  2. Recibido: Confirmado pero no pagado.
  3. En preparación: Pagado y en cola de cocina.
  4. Cocinando: Un cocinero ha comenzado la elaboración.
  5. Listo cocina: Elaboración terminada.
  6. Terminado: Listo para recogida (camarero ha completado con bebidas, etc.).
  7. Entregado: El cliente lo ha recibido.

Incluye tipos de pedido (Local/Llevar), simulación de pago con tarjeta y un historial de pedidos en el perfil del cliente.

Funcionalidad 3: Preparación de Pedidos (Cocina)

Interfaz diseñada para tablets en formato apaisado para los cocineros. Permite:

  • Asignarse un pedido en estado "En preparación" (pasa a "Cocinando").
  • Confirmar la preparación de productos individualmente.
  • Finalizar el pedido para avisar a sala.

El Gerente dispone de una vista global donde puede ver qué cocinero (mediante su avatar) está preparando cada pedido.
